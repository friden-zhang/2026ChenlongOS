## 1. SPI 协议是什么
SPI(Serial Peropheral interface, 串行外设接口)是一种高速的、全双工、同步的通信总线，被广泛应用于微控制器(MCU)与各种外围设备(如传感器、Falsh等)之间的短距离通信。


## 2. SPI协议详解
### 2.1 硬件与引脚

SPI 通常需要 4 根线来进行通信（如果是单向传输也可以精简为 3 根线）。它采用主从模式（Master-Slave），通信总是由主设备发起。
这四根线分别是：
- SCLK / SCK (Serial Clock): 串行时钟线。由主设备（Master）产生，用于同步数据传输。
- MOSI (Master Out Slave In): 主设备输出 / 从设备输入线。数据从主设备发送到从设备。
- MISO (Master In Slave Out): 主设备输入 / 从设备输出线。数据从从设备发送到主设备。
- CS / SS (Chip Select / Slave Select): 片选信号线。由主设备控制，通常是低电平有效。当主设备将这条线拉低时，表示选中了该从设备，通信开始。通过提供多个 CS 引脚，一个主设备可以连接多个从设备。

### 2.2 数据传输原理
SPI 的本质是移位寄存器（Shift Register）的环形连接。
主设备和从设备内部各有一个移位寄存器。在 SCLK 时钟信号的驱动下，主设备的寄存器内容和从设备的寄存器内容相当于连成了一个环。
每产生一个时钟周期，主设备将自己寄存器的最高位（或最低位）通过 MOSI 发送给从设备，同时从设备也将自己寄存器的位通过 MISO 发送给主设备。经过 8 个时钟周期后，主从设备正好完成了一个字节的数据交换。因此，SPI 的发送和接收是同时进行的（全双工）。

### 2.3 SPI 的四种工作模式
不同的设备可能在不同的时钟状态下采集和改变数据。SPI 定义了两个配置参数：
- CPOL (Clock Polarity, 时钟极性): 决定 SCLK 在空闲状态（没有数据传输时）是高电平还是低电平。
- - CPOL = 0：空闲时 SCLK 为低电平。
- - CPOL = 1：空闲时 SCLK 为高电平。

- CPHA (Clock Phase, 时钟相位): 决定数据是在时钟的第一个跳变沿（上升沿或下降沿）采样，还是在第二个跳变沿采样。
- - CPHA = 0：在时钟的第一个跳变沿进行数据采样。
- - CPHA = 1：在时钟的第二个跳变沿进行数据采样。

这两个参数组合起来构成了 SPI 的四种模式（Mode 0 ~ Mode 3）。最常用的是 Mode 0（CPOL=0, CPHA=0）和 Mode 3（CPOL=1, CPHA=1）。 通信前，主从设备的模式必须设置一致。

### 2.4 优缺点
优点: 传输速度快（通常可达十几甚至几十 MHz）；全双工通信；硬件协议简单，无需像 I2C 那样发送设备地址。
缺点: 没有应答机制（不知道对方是否真的收到了）；引脚相对较多（每个额外的从设备都需要多一根 CS 线）；只适合板级短距离通信。


## 3. 仿真
> 使用 Wokwi/PlatformIO 进行仿真，支持 VSCode 插件安装，通过 [视频](https://www.youtube.com/watch?v=ECNTyMm_5PE)、[文档](https://docs.wokwi.com/vscode/getting-started) 了解Wokwi基本使用方式

``` bash
cd lecture-0/spi-sim
pio project init --board uno
```

使用 `Arduino Uno` 作为 SPI 主设备，驱动一块 `ILI9341` TFT 屏幕。

### 3.1 接线

这个仿真里实际用到的线并不多，核心就是把 Uno 的硬件 SPI 引脚接到屏幕上：

- `5V -> VCC`
- `GND -> GND`
- `D10 -> CS`
- `D9 -> D/C`
- `D11 -> MOSI`
- `D13 -> SCK`

这里我没有接 `MISO`，因为这个实验只需要把图形数据从 `Uno` 发到屏幕，不需要从屏幕把数据再读回来。

![](https://my-obsidian-vault-1419031144.cos.ap-beijing.myqcloud.com/images/20260418234548120.png)

上面这张图就是接线完成后的样子。可以看到，`D10 / D11 / D13` 这几个和 SPI 相关的数字引脚都用上了，其中：

- `D10` 负责片选 `CS`
- `D11` 是 `MOSI`
- `D13` 是时钟 `SCK`
- `D9` 虽然不是 SPI 硬件专用脚，但这里作为屏幕的 `D/C` 控制脚使用，用来区分“当前发过去的是命令还是显示数据”

### 3.2 代码做的事情

`lecture-0/spi-sim/src/main.cpp` 里主要做了下面几件事：

> Code commit: 3b24c41bb7cb0c1d5bdfc8bd660f94b238b2d150

1. 初始化 `Adafruit_ILI9341` 驱动对象。

这个程序本质上是在不断调用图形库，把绘图命令和像素数据通过 SPI 一直写给 `ILI9341` 控制器。

### 3.4 波形观察

![](https://my-obsidian-vault-1419031144.cos.ap-beijing.myqcloud.com/images/20260419000113679.png)

为了观察 SPI 的实际时序，我在 Wokwi 电路里又加了一个 `Logic Analyzer`。这里的连线关系是：

- `D0 -> SCK`
- `D1 -> MOSI`
- `D2 -> CS`
- `D3 -> D/C`
- `D7 -> Trigger`

其中前四根线对应的是我真正关心的 SPI 相关信号，`D7` 不是屏幕本身的连接线，而是我额外加的一个触发信号。程序在准备发送那一小段 SPI 数据之前，会先把 `D7` 拉高；发送结束后，再把它拉低。这样逻辑分析仪就能更集中地抓到“真正有用的那一小段传输”，而不是把整段仿真时间都录下来。

![](https://my-obsidian-vault-1419031144.cos.ap-beijing.myqcloud.com/images/20260419001012986.png)

第二张图是把导出的 `spi-capture.vcd` 打开之后看到的波形。这个视图一开始还是比较“粗”的缩放比例，所以看到的是一个整体时间窗口，而不是位级别的细节。不过即使在这个尺度下，也已经能看出几个关键点：

- `D7` 中间有一段明显的高电平，这一段就是我主动标记出来的“抓取窗口”。
- 波形真正有意义的 SPI 传输，就集中在这段窗口附近。
- 因为当前缩放得比较远，`SCK` 和 `MOSI` 这种高速翻转信号被压缩在很窄的一段时间里，所以在这张图上看起来不明显。

如果继续在波形软件里把 `D7` 为高的那一小段放大，就可以更清楚地看到下面这些关系：

![](https://my-obsidian-vault-1419031144.cos.ap-beijing.myqcloud.com/images/20260419001320394.png)

这张图已经比上一张放大了很多，所以可以开始直接看出 SPI 几根线之间的配合关系：

- 最上面的 `D0` 是 `SCK`，可以看到它不是一直跳变，而是以“一串脉冲、一段空闲、再一串脉冲”的形式出现。这说明主控是在分几次把命令或数据发给屏幕，而不是一上来就连续不停地灌数据。
- `D1` 是 `MOSI`，它会在 `SCK` 脉冲期间发生变化，用来承载真正送往屏幕控制器的数据位。
- `D2` 是 `CS`，这里大部分时间保持有效，说明这一小段放大窗口里屏幕一直处于被选中状态；到尾部出现一次释放，表示这一段 SPI 传输告一段落。
- `D3` 是 `D/C`，它在图里有几次明显的高低切换，这正好说明屏幕通信里并不是所有字节都属于同一种类型：有些字节是命令，有些字节是数据。这里的 `D/C` 就是 `Data / Command`，也就是“当前这一拍发过去的是数据还是命令”的选择信号。

如果把这张图和前面两张图连起来看，理解会更顺一些：

1. 第一张图说明了分析仪是怎么接进去的。
2. 第二张图说明了整段抓取窗口大概落在哪个时间范围。
3. 第三张图则开始展示位级别附近的真实细节，也就是 `SCK`、`MOSI`、`CS`、`D/C` 是如何一起工作的。

从这个放大波形里，至少可以比较直观地看出 SPI 写屏时的两个特点：

- 时钟 `SCK` 是整个传输的节拍基准，只有在真正发送数据时才会出现成串脉冲。
- 对屏幕这类外设来说，除了标准的 `CS / SCK / MOSI` 之外，`D/C` 这种额外控制脚也很重要，因为它决定了当前这批字节究竟应该被解释成“命令”还是“显示数据”。

这个实验里没有连接 `MISO`，所以波形中也不会看到从设备返回数据的那一路。这一点也正好对应了当前这个场景：`Uno` 只是把显示命令和像素数据写给 `ILI9341`，并没有从屏幕读回内容。

从波形观察的角度看，这个实验至少说明了两件事：

1. `SPI` 的几根关键信号线确实是在协同工作的，而不是只有某一根线单独变化。
2. 对显示屏这种外设来说，主控往往以“片选 + 时钟 + 数据”的组合方式持续写入内容，`D/C` 这样的附加控制脚则负责告诉外设“当前发的是命令还是显示数据”。
